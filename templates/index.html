Relaxed Functional Dependencies

In relational databases, a Relaxed Functional Dependency (RFD) is an integrity constraint X -> Y between two sets of attributes X and Y, meaning that if two tuples have similar value on X, then they must have similar values on Y. X is named Left Hand Side (LHS), while Y is the Right Hand Side (RHS). Two values of an attribute are similar if their distance is lower than the similarity threshold defined on that attribute. The function to assess similarity is edit distance for strings and difference for numbers.

For example, consider the following dataset:

# Model Year Price # Owners
1 Renault Clio 2018 8000 1
2 Renault Clio 1999 1200 1
3 Renault Clio dci 2019 8500 1
4 Ford Fiesta Tdci 2022 10000 1

The RFD Model≤4, Year≤1 -> Price≤500 holds on this dataset, since similar models with similar years have a similar price (4, 1,and 500 are the similarity thresholds). In fact, the first and the third tuple (rows) have similar values for the attributes Model and Year and they also have values for the attribute Price that differ for less than the similarity threshold.

After the insertion of a new tuple, an existing RFD can be invalidated only if the new tuple has similar values on the LHS with respect other tuples but it has different values on the RHS. In this case, a specialized RFD with additional attributes on the LHS may be valid on the dataset.

For example, suppose that we add tuple 5 to the previous dataset:

# Model Year Price # Owners
1 Renault Clio 2018 8000 1
2 Renault Clio 1999 1200 1
3 Renault Clio dci 2019 8500 1
4 Ford Fiesta Tdci 2022 10000 1
5 Renault Clio dci 2019 5500 3

The RFD Model≤4, Year≤1 -> Price≤500 does not hold anymore, since tuple 5 has a similar model and a similar year with respect to tuple 1 and 3, but has a different price.

However, the invalidated RFD can be specialized by the RFD Model≤4, Year≤1, #Owners≤0 -> Price≤500 which is valid on the updated dataset. In fact, only tuple 1 and tuple 3 satisfy the constraints on the left-hand side, and they have similar values for the attribute on the right-hand side.

Intead, after the deletion of a tuple, a generalized RFD with a subset of attributes on the LHS may be valid on the dataset if the deleted tuple had similar values on the reduced LHS with respect other tuples but it had different values on the RHS. In this case, the previously holding RFD is no longer minimal, and replaced by the generalized RFD.

For example, suppose that we delete the tuple we added in the previous example:

# Model Year Price # Owners
1 Renault Clio 2018 8000 1
2 Renault Clio 1999 1200 1
3 Renault Clio dci 2019 8500 1
4 Ford Fiesta Tdci 2022 10000 1
5 Renault Clio dci 2019 5500 3

As discussed, tuple 5 was invalidating the original RFD Model≤4, Year≤1 -> Price≤500, leading to the evolution of this dependency with the specialized RFD Model≤4, Year≤1, #Owners≤0 -> Price≤500.

Thus, the deleted tuple had similar values for the original LHS (Model,Year) but a different price. After the deletion, the specialized RFD Model≤4, Year≤1, #Owners≤0 -> Price≤500 is no longer minimal, since the RFD Model≤4, Year≤1 -> Price≤500 is valid and has a subset of its attributes on the LHS.

The Ryan Tool

The RFD explainabilitY & ANalysis tool (RYAN) tool is one of the first AI-based tools that analyzes how rfds discovered on a dataset evolve as the data changes. RYAN is able to combine interactive visual components and emergent LLMs to provide explanations about the evolution of discovered RFDs after data changes (insertions, deletions and updates).

Figure 1: Homepage. From here, the user can load the dataset (in csv format) and the discovery results (in json format).

After loading (see Figure 1) the dataset and the results of two RFD discovery processes performed at different time steps, Ryan provides different visual components to analyze them:

  • The first visual component is the RFD tree (Figure 2). This tree introduces a collapsible representation of all RFDs discovered on the updated data, i.e., at time ùúè + 1, and their relation with those discovered at time ùúè. The RFD tree divides the discovered RFDs into five main categories: Generalizations, Specializations, RFDs found, RFDs not found, and New RFDs. The rfd tree is composed of a root node, labeled ‚ÄúRFDs‚Äù, linked to five child nodes, one for each of the categories described above. The Generalizations/Specializations represent all the rfds discovered at time ùúè + 1 that generalize/specialize at least one rfds discovered at time ùúè . The rfds found nodes are all valid rfds both at time ùúè and ùúè + 1, whereas the rfds not found nodes are all rfds that were discovered at time ùúè but were not discovered in any forma at time ùúè + 1. The New rfds node represents all rfds discovered at the time ùúè + 1 that were not discovered at the time ùúè . By clicking on one of these nodes, the tree can show new nodes, each labeled with an RHS attribute (Figure 3). By clicking further on one of these nodes (Figure 4), the tree shows its children, which are labeled with the LHS attributes associated with the chosen RHS, thus showing all rfds.
Figure 2: RFD tree and Sunburst. After uploading the files, RYAN displays two visual components: an RFD tree (left) and a sunburst chart (right)

Figure 3: RFD tree navigation. In this case the user selects the specialized RFDs, and the tree displays the different RHS of all the RFDs in this category.

Figure 4: RFD tree navigation (2). By clicking on one of the RHS, RYAN will display the LHS of all the dependencies that have the selected RHS in the selected category.

  • The second visual component is the Sunburst chart (right side of Figures 2, 3, 4). Similar to the rfds tree, the Sunburst component provides a hierarchical representation of rfds, considering a higher level of abstraction to provide users with statistics about the five categories of rfds. The innermost ring of the rfds Sunburst, labeled ‚ÄúRFDs‚Äù, is the root of the chart. The second ring contains the segments representing the categories of rfds. Finally, the third and fourth rings contain the segments of the RHS and respective LHS of the rfds discovered upon data changes, respectively.
  • The third visual component is the Comparative Datatable (Figure 5). It provides a more detailed representation of rfds and allows users to get a precise explanation of the impact that data changes have had on the holding rfds. It is composed of five different columns: the first two represent the LHS and the RHS of an rfd ùúô valid at time ùúè ; the third is the motivation with which the rfd discovered at the time ùúè + 1 is related to ùúô, while the last two contains the attributes on the LHS and the RHS of an rfd validated at time ùúè + 1. When displaying a specialized dependency, the additional attributes are highlighted in green, whereas when displayng a generalized dependency, the removed attributes are highlighted in red (Figure 6). As regards the notation, each attribute and the corresponding threshold are written as follows: AttributeName_threshold, were the _ indicates the ≤ constraint.
Figure 5: Comparative Datatable. After fully expanding a node in the RFD tree, the datatable will display the dependencies belonging to that node. If the selected dependency is a specialization or a generalization of a previous RFD discovered before data updates, the table will display the original dependency on the same row.

Figure 6: The comparative datatable for generalized RFDs.

In case of an RFD that is a specialization or a generalization of an RFD discovered before data updates, the user can click on the EXPLAIN button displayed in the associated row of the datatable. This function allows to provide an explanation of why the RFD evolved by analyzing the data updates occurred. RYAN will compile a prompt containing the relevat information necessary to explain the evolution and will provide it to an LLM. Once ready, the answer will be shown to the user in a new window (see Figure 7).

Figure 7: The Explain feature of RYAN.

In addition, the datatable provides also the INTERACT feature. By clicking on the associated button, a new window containing the compiled prompt will be displayed. Here, the user can customize the prompt, or ask for additional questions. Once the customized prompt is ready, the user can click on the Ask button and wait for the answer.

Figure 8: The Interact feature of RYAN.
